# 核心业务链路

> 核心业务流程的文档化描述。Agent 理解 Harness 框架运行上下文的基础素材。

---

## 链路 1：变更全生命周期（6 阶段流程）

### 触发条件
- 用户提出开发需求（新功能 / 修 Bug / 重构）
- Agent 启动新变更处理

### 完整链路

```
[用户需求]
    │
    ▼
┌──────────────────────────────────────────────┐
│  阶段 1：需求分析                              │
│  Owner Agent 阅读需求 → 生成 spec.md           │
│  输出：(1) spec.md  (2) tasks.md              │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│  阶段 2：需求评审                              │
│  Evaluator Agent 评审 spec.md                 │
│  ├── 完整性检查（需求/范围/验收标准）            │
│  ├── 风险分析                                  │
│  └── 批准/拒绝                                 │
│  输出：review-v1.md                            │
└──────────────────┬───────────────────────────┘
                   │  批准
                   ▼
┌──────────────────────────────────────────────┐
│  阶段 3：编码                                  │
│  Owner Agent 按 tasks.md 编码                 │
│  遵守架构规则（arch-rules.md）+ 编码规范        │
│  输出：代码文件                                │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│  阶段 4：代码评审                               │
│  Evaluator Agent 评审代码                     │
│  ├── 正确性 / 安全性 / 性能                    │
│  ├── 单文件 ≤250 行                            │
│  ├── 无硬编码 / 无类型擦除                     │
│  └── 批准/拒绝                                 │
│  输出：review-v2.md                            │
└──────────────────┬───────────────────────────┘
                   │  批准
                   ▼
┌──────────────────────────────────────────────┐
│  阶段 5：测试 + 质量门禁                        │
│  QG-1 编译 → QG-2 类型 → QG-3 风格 → QG-4 复杂度│
│  QG-5 长函数 → QG-6 硬编码 → QG-7 测试 → QG-8 覆盖│
│  输出：test-plan.md + qg-result-*.md           │
└──────────────────┬───────────────────────────┘
                   │  全部 pass
                   ▼
┌──────────────────────────────────────────────┐
│  阶段 6：交付归档                               │
│  更新 summary.md → 完成进度 → 清理临时文件       │
│  输出：归档的变更目录                            │
└──────────────────────────────────────────────┘
```

### 角色与职责

| 角色 | Agent 类型 | 职责 | 参与阶段 |
|------|-----------|------|----------|
| Owner Agent | 主执行 Agent | 需求分析、编码、归档 | 1, 3, 6 |
| Evaluator Agent | 评审 Agent | 评审批准/拒绝 | 2, 4, 5 |
| Initializer Agent | 初始化 Agent | 环境搭建 | 阶段前 |

### 质量门禁

每个变更通过前必须通过全部 8 项质量门禁（QG-1 ~ QG-8），由 `verify-qg.sh` 或 `verify-qg.py` 执行。详见 `verify-qg.sh` 中的 QG-1~8 实现。

## 链路 2：检测与适配流程

### 触发条件
- Agent 新会话启动（`init.sh`）
- 运行平台不确定时

### 完整链路

```
[Agent 会话启动]
    │
    ┌▼──────────────────────────────────────────┐
    │  init.sh（会话启动脚本）                     │
    │                                            │
    │  1. detect-platform.sh                     │
    │     → 检测 ${PLATFORM}                     │
    │        (opencode/claude/codex/cursor/ci)   │
    │     → 设置工具别名（read/write/shell 命令）  │
    │     → 定位 MCP 配置路径                     │
    │                                            │
    │  2. detect-build.sh                        │
    │     → 检测 ${BUILD_TOOL}                   │
    │        (maven/gradle/pip/npm/cargo/go)     │
    │     → 设置 ${BUILD_CHECK_CMD}              │
    │                                            │
    │  3. 运行编译检查                            │
    │     → 成功/失败报告                         │
    │                                            │
    │  4. 检查变更目录                            │
    │     → 显示进行中的变更数                     │
    └────────────────────────────────────────────┘
```

### 平台检测决策树

```
[当前进程]
    │
    ├── 存在 $CLAUDE_CODE=1           → Claude Code
    ├── 存在 $OPENAI_CODEX=1          → Codex CLI
    ├── 进程名含 codex                 → Codex CLI
    ├── 存在 $GITHUB_ACTIONS=true     → GitHub Actions (CI)
    ├── 存在 ~/.config/opencode/      → OpenCode
    ├── 存在 .opencode/ 或 opencode.jsonc → OpenCode
    ├── 存在 .vscode/                 → Cursor (低置信度)
    └── 其他                           → generic（通用模式）
```

### 构建检测决策树

```
[项目根目录]
    │
    ├── 存在 pom.xml          → maven
    ├── 存在 build.gradle     → gradle
    ├── 存在 pyproject.toml   → pip
    ├── 存在 package.json     → npm
    ├── 存在 Cargo.toml       → cargo
    ├── 存在 go.mod           → go
    └── 其他                  → unknown（降级为仅检查）
```

## 链路 3：熵清理流程（GC）

### 触发条件
- 定期自动运行（推荐每周一次）
- 变更归档后触发
- 用户手动执行 `gc-scan.sh`

### 完整链路

```
[gc-scan.sh 执行]
    │
    ▼
┌──────────────────────────────────────────────┐
│  类别 1：注释漂移                              │
│  grep 注释与实现不一致的场景                     │
│  例如：TODO 未清、过期注释                      │
└──────────────────┬───────────────────────────┘
                   ▼
┌──────────────────────────────────────────────┐
│  类别 2：死代码 / 注释掉的代码                   │
│  搜索被注释的生产代码（非文档注释）              │
└──────────────────┬───────────────────────────┘
                   ▼
┌──────────────────────────────────────────────┐
│  类别 3：过期引用                               │
│  发现引用不存在的文件、目录或配置项              │
└──────────────────┬───────────────────────────┘
                   ▼
┌──────────────────────────────────────────────┐
│  类别 4：测试烂掉（Test Rot）                   │
│  跳过/忽略的测试、无断言的测试                  │
└──────────────────┬───────────────────────────┘
                   ▼
┌──────────────────────────────────────────────┐
│  类别 5：架构漂移                               │
│  实现与架构文档不符的模式                      │
└──────────────────┬───────────────────────────┘
                   ▼
┌──────────────────────────────────────────────┐
│  结果处理                                      │
│  ├── 无发现 → exit 0                          │
│  ├── 有发现 → 列出各项 → exit 1（CI 会失败）   │
│  ├── --json  → JSON 格式输出                   │
│  └── --category N → 仅扫描指定类别              │
└──────────────────────────────────────────────┘
```

### GC 频率建议

| 类别 | 建议频率 | 自动修复 | 说明 |
|------|---------|----------|------|
| 注释漂移 | 每次变更 | 部分可自动 | 低风险 |
| 死代码 | 每周 | Agent 判断 | 需人工确认 |
| 过期引用 | 变更归档后 | 自动 | 低风险 |
| 测试烂掉 | 每次测试 | Agent 修复 | 中风险 |
| 架构漂移 | 每月 | Agent 评估 | 需评审 |

## 链路 4：CI 自动化流程

### 触发条件
- `git push` 到 `main` 分支
- PR 创建/更新

### 完整链路

```
[git push origin main]
    │
    ▼
GitHub Actions Workflow: harness-ci.yml
    │
    ├── Job 1: init
    │   ├── detect-build.sh  → 设置 BUILD_TOOL
    │   ├── detect-platform.sh → 设置 PLATFORM=github-actions
    │   └── 缓存 restore
    │
    ├── Job 2: quality-gates (depends on init)
    │   ├── QG-1: Compile Check    ($BUILD_CHECK_CMD)
    │   ├── QG-2: Type Check       (tsc/basedpyright)
    │   ├── QG-3: Code Style       (formatter check)
    │   ├── QG-4: Complexity       (radon cc / lizard)
    │   ├── QG-5: Long Functions   (radon raw)
    │   ├── QG-6: Hardcoded Config (grep)
    │   ├── QG-7: Test Execution   (test framework)
    │   └── QG-8: Coverage         (coverage report)
    │
    ├── Job 3: lint-and-shellcheck
    │   ├── ShellCheck (所有 .sh)
    │   └── yamllint (所有 .yml .yaml)
    │
    └── Job 4: docs-integrity
        ├── 检查所有 .md 链接有效性
        └── 验证 template 完整性
```

## 高频变更区（Hot Paths）

以下链路是框架中变更频率最高的区域，修改时需特别注意：

| 变更区域 | 频率 | 典型变更类型 | 注意事项 |
|----------|------|-------------|----------|
| 质量门禁脚本（verify-qg.*） | 高 | 新增检查项、调整阈值 | 保持 bash/Python 双实现同步 |
| Agent 流程（owner-agent.md） | 中 | 调整阶段、优化编排 | 需同步 evaluator-agent.md |
| 平台适配（platform.md） | 中 | 新增平台支持 | 需同步 detect-platform.sh 和 mcp/ |
| 变更模板（template/） | 中 | 新增/优化模板 | 需同步 init.md 的验证清单 |
| CI 配置 | 低 | 调整 pipeline | 注意 keep 最新的 GitHub Actions 语法 |
