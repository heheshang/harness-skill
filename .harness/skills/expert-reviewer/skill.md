# Expert Reviewer Skill

> 专家评审技能。用于需求评审和编码评审阶段。
>
> **评审检查清单**已在 `agents/evaluator-agent.md` 中完整定义，此处只保留评审方法和报告模板。
> 避免重复：evaluator-agent.md 是检查清单的 Single Source of Truth。

## 评审类型

三种评审由 Evaluator Agent 执行，Owner Agent 在对应阶段调用：

| 评审类型 | 调用阶段 | Evaluator Agent 检查维度 |
|----------|----------|------------------------|
| 计划评审 | 阶段 2 | 需求完整性、正确性、变更范围、影响分析、验收标准 |
| 执行评审 | 阶段 4 | 业务正确性、变更完整性、异常处理、边界条件、安全合规 |
| 测试评审 | 阶段 5 | 测试覆盖率、正确性、边界测试、Mock、独立性 |

完整检查清单参见 `agents/evaluator-agent.md`。

## 评审方法

### 评审视角

- Evaluator Agent 需要使用不同于编码 Agent 的检查视角
- 假设代码是"有问题的"，主动寻找缺陷而非被动确认正确
- 站在"如果这个代码上线了会怎样"的角度思考

### 每轮评审产出

1. 读取目标文件（spec.md / 源码 / 测试）
2. 逐项检查 Evaluator Agent 检查清单
3. 记录每条问题：描述 + 修改建议 + 优先级
4. 汇总评审结论

## 评审报告模板

```markdown
# 评审报告（第 {n} 轮）

## 评审信息
- **需求名称**：{需求名称}
- **评审类型**：{计划评审 / 执行评审 / 测试评审}
- **评审时间**：{YYYY-MM-DD HH:mm}
- **评审结论**：{通过 / 需修改}

## 评审意见

### MUST FIX（必须修改）
1. {问题描述}
   - **位置**：{文件路径:行号}
   - **建议**：{修改建议}
   - **理由**：{为什么这是必须修复的}

### LOW（建议修改）
1. {问题描述}
   - **位置**：{文件路径:行号}
   - **建议**：{修改建议}

### INFO（仅供参考）
1. {观察意见}
   - **说明**：{详细说明}

## 总结
- MUST FIX: {数量}
- LOW: {数量}
- INFO: {数量}
```

## 评审原则

| 原则 | 说明 |
|------|------|
| 每条意见可操作 | 必须包含：**问题描述 + 修改建议 + 优先级** |
| 优先级明确 | MUST FIX（阻塞）/ LOW（建议）/ INFO（观察） |
| 位置精确 | 评审意见必须标注到文件和行号 |
| 不跳问题 | 发现 MUST FIX 必须记录，不能"为了通过评审而忽略" |
