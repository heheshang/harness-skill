# Request Analysis Skill

> 需求分析技能。用于需求分析阶段，将模糊需求转化为结构化规格说明。

## 输入
- 需求描述（来自用户）
- 项目上下文（来自 `.harness/wiki/`，可选）

## 输出
- `spec.md` — 需求规格说明书

## 执行步骤

### Step 1：理解需求
1. 仔细阅读需求描述，识别核心目标
2. 区分"要什么"和"怎么实现"——需求分析只关注前者
3. 识别需求的边界（什么在范围内，什么不在范围内）

### Step 2：识别变更范围
1. 确定需求涉及的功能模块
2. 确定需求涉及的代码层（Controller / Service / Domain / DAO / Adapter）
3. 识别可能受影响的关联模块和链路

### Step 3：影响分析
1. **上下游影响**：接口变更是否影响下游调用方
2. **数据影响**：数据模型变更是否涉及数据迁移或兼容性
3. **配置影响**：是否需要新增或修改配置项
4. **国际化影响**：新增字段或提示是否需要国际化
5. **性能影响**：是否会引入性能瓶颈（如新增热点缓存）

### Step 4：编写验收标准
1. 每个验收标准必须可验证（Given-When-Then 格式）
2. 覆盖正常场景和异常场景
3. 覆盖边界条件

### Step 5：输出 spec.md

#### spec.md 模板

```markdown
# 需求规格说明书

## 背景
{描述需求的业务背景和动机}

## 需求描述
{清晰描述需求的核心内容}

## 变更范围

### 涉及的模块
- {模块名}：{变更内容简述}
- {模块名}：{变更内容简述}

### 涉及的代码层
- [ ] Controller / REST
- [ ] Service
- [ ] Domain
- [ ] DAO / Mapper
- [ ] Adapter / RPC

### 不涉及的模块
- {模块名}：{原因}

## 影响分析

| 影响维度 | 分析 | 风险等级 |
|----------|------|----------|
| 上下游兼容性 | {分析} | {高/中/低} |
| 数据兼容性 | {分析} | {高/中/低} |
| 配置变更 | {分析} | {高/中/低} |
| 国际化 | {分析} | {高/中/低} |
| 性能 | {分析} | {高/中/低} |

## 验收标准

### 正常场景
1. Given {前置条件}, When {操作}, Then {期望结果}
2. Given {前置条件}, When {操作}, Then {期望结果}

### 异常场景
1. Given {异常条件}, When {操作}, Then {期望错误处理}
2. Given {异常条件}, When {操作}, Then {期望错误处理}

### 边界条件
1. {边界情况}: {期望行为}

## 备注
{其他需要说明的事项}
```

## 质量门禁（自检）
- [ ] 需求背景和动机清晰
- [ ] 变更范围明确标记了涉及的模块和代码层
- [ ] 影响分析覆盖了所有相关维度
- [ ] 验收标准可验证（非模糊描述）
- [ ] 验收标准覆盖正常/异常/边界场景
- [ ] 不包含"可能"、"大概"、"也许"等不确定词汇
