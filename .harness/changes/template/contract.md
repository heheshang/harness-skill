# Sprint Contract（冲刺合同）

> Generator-Evaluator 双方法成一致的"完成定义"。
> 在编码开始之前签署，避免在实现过程中各自理解"done"。

参考：[Anthropic: Harness design for long-running application development]

## 1. 基本信息

| 字段 | 值 |
|------|------|
| 特性 ID | F-001 (对应 feature_list.json) |
| 特性描述 | （一句话） |
| Generator | Agent Name |
| Evaluator | Agent Name |
| 签署时间 | YYYY-MM-DD |
| 迭代次数 | 1 / 2 / 3 |

## 2. Generator 提案

### 2.1 要构建什么

- 列出具体要增/删/改的代码
- 涉及的层、接口、模型
- 不做的范围（明确排除）

### 2.2 成功验证方式

- 如何验证这项功能正常工作
- 自动化测试策略
- 手工验证步骤（如有）

### 2.3 评估风险

- 兼容性：是否向后兼容
- 性能：是否引入性能影响
- 安全：是否涉及敏感操作

## 3. Evaluator 审查

### 3.1 提案评估

| 维度 | 评分 (0-10) | 阈值 | 通过? |
|------|-------------|------|-------|
| 范围明确性 | | ≥ 7 | ✅ / ❌ |
| 可验证性 | | ≥ 7 | ✅ / ❌ |
| 风险评估 | | ≥ 6 | ✅ / ❌ |
| 整体可行性 | | ≥ 7 | ✅ / ❌ |

### 3.2 审查意见

- 需要澄清的点
- 需要补充的验证方式
- 需要增加的测试范围

### 3.3 评审结论

- [ ] 通过（无需修改）
- [ ] 需修改（详见上节）

## 4. 最终协议

```
签署条件：
1. Evaluator 所有维度评分 >= 阈值
2. 合同中明确记载了"done"的客观证据
3. Generator 和 Evaluator 认知一致

不签署的后果：
- 不进入编码阶段
- 回到提案阶段修改
- 超 3 轮未签署 → 升级到 Owner Agent 仲裁
```
