# 流程规则

> 10 阶段开发流程的**规则**，而非阶段定义。
> **阶段定义**在 `agents/owner-agent.md` 中，这里只定义**流程规则与约束**。

## 规则 0：会话必须先启动（Session Startup Ritual）

> 参考 owner-agent.md 第零节「会话启动仪式」

**每次新 Agent 会话开始时，必须先完成启动流程：**

```
0.1 执行 init.sh（如存在）
0.2 读取 git log 了解最近进度
0.3 读取 feature_list.json 或 summary.md 恢复上下文
0.4 运行编译检查确认代码库未被破坏
0.5 然后才进入阶段 1 需求分析
```

**违反了这条规则，就像走进手术室却不洗手。**

## 规则 1：阶段不可跳过

无论需求大小，必须按顺序经过完整的 N 阶段流程。
简单需求的每个阶段执行时间会自然缩短，但流程的一致性保证了不会因为"这次改动很小"就跳过关键环节。

**例外**：经人工授权的阶段可合并或跳过，但必须在 `summary.md` 中记录原因。

## 规则 2：每个阶段有明确的三要素

| 要素 | 定义 | 示例 |
|------|------|------|
| Entry Criteria | 什么条件满足时可以进入 | 上阶段质量门禁通过 + Human-in-the-Loop 确认（如需） |
| Skill Injection | 需要加载哪个技能包 | 编码阶段加载 `.harness/skills/coding-skill/` |
| Quality Gate | 产出必须满足什么条件 | QG-5: status == SUCCESS && total > 0 && passed == total |

## 规则 3：精确的失败回退路径

```
CI 失败 (测试为 0/0) ─────────→ 阶段 5（测试编写）
CI 失败 (编译错误) ──────────→ 阶段 3（编码实现）
CI 失败 (测试失败) ──────────→ 阶段 5（修复测试）
集成测试失败 ────────────────→ 阶段 3/5（根据失败原因）
需求不符 ──────────────────→ 阶段 1（需求分析）
评审超轮次上限 ─────────────→ 升级到人工决策
```

## 规则 4：评审轮次上限

| 评审类型 | 最大轮次 | 超出后的处理 |
|----------|----------|--------------|
| 需求评审 | 3 轮 | 升级到人工决策 |
| 编码评审 | 2 轮 | 升级到人工决策 |
| 测试评审 | 2 轮 | 升级到人工决策 |

## 规则 5：Human-in-the-Loop 确认点

流程中嵌入 5 个强制人工确认节点：

| 编号 | 确认点 | 位置 | 说明 |
|------|--------|------|------|
| H1 | 需求规格确认 | 阶段 1 → 阶段 2 | spec.md 内容是否符合预期 |
| H2 | 需求评审通过 | 阶段 2 → 阶段 3 | 需求评审通过，确认进入编码 |
| H3 | 编码评审通过 | 阶段 4 → 阶段 5 | 编码质量确认 |
| H4 | 部署参数确认 | 阶段 7 → 阶段 8 | 部署环境/版本参数确认 |
| H5 | 交付确认 | 阶段 10 | 全流程交付确认 |

## 规则 6：文档同步约束

- 代码变更后必须同步检查 Wiki 知识库是否需要更新
- 接口变更必须同步更新接口文档
- 数据模型变更必须同步更新 data-model.md
- 评审文件旧版本永不删除（review-v1.md, review-v2.md...）

## 规则 7：编码前置条件

- 不在未经评审通过书面计划（spec.md + tasks.md）之前写代码
- 编码前必须先理解现有代码逻辑
- 一切修改前先读取目标文件的现有内容

## 规则 8：阶段精简原则

**需求越简单，阶段执行时间越短，但阶段数量不改。**
10 个阶段分为两部分：

| 范围 | 阶段 | 说明 |
|------|------|------|
| **核心 6 阶段** | 1→6 | Agent 可在单次会话中独立完成 |
| **扩展 4 阶段** | 7→10 | 涉及部署/运维，通常在 Agent 会话外完成 |

Agent 在单个 session 中至少完成核心 6 阶段。
扩展 4 阶段需要人工确认和跨 Session 执行。
