# 质量门禁定义

> 可程序化验证的质量检查条件。一切不可被机器验证的约束，在 Agent 执行中都是无效约束。
>
> **核心原则**：If it can't be mechanically enforced, the agent will drift.

## 门禁检查清单

### QG-1：需求规格完整性

```
条件：
  - spec.md 文件存在于变更目录下
  - spec.md 包含以下必填章节：
    [x] 背景
    [x] 需求描述
    [x] 变更范围
    [x] 影响分析
    [x] 验收标准
    [x] 备注（必填但允许"无"）
  - 影响范围内涉及的所有模块已明确列出
  - 无模糊描述（不包含"可能"、"大概"、"也许"、"maybe"、"probably"等不确定词汇）
验证方式：文件存在性检查 + 关键词匹配
```

### QG-2：评审通过性

```
条件：
  - 评审报告文件存在（review-v{n}.md）
  - 评审结论为"通过"（不包含 MUST FIX 级别意见）
  - 若存在多个评审报告，以最新版本为准
  - 评审轮次 ≤ 3（需求评审）/ ≤ 2（编码评审/测试评审）
验证方式：文件存在性检查 + 内容关键词匹配
```

### QG-3：编译检查

```
条件：
  - 项目编译无错误
  - 项目编译无警告（WARNING 级别以上为 0）
验证方式：detect-build.sh 自动检测后执行 ${BUILD_CHECK_CMD} 返回值检查
```

### QG-4：单元测试门禁

```
条件：
  - total_tests > 0         （测试用例数大于 0）
  - passed == total_tests   （全部通过）
  - time < threshold        （执行时间在合理范围内）
验证方式：
  - 解析 CI 返回结果：
    status == SUCCESS
    && total_tests > 0
    && passed == total_tests
```

### QG-5：CI 门禁

```
条件：
  - CI pipeline status == SUCCESS
  - 测试阶段 total_tests > 0（防止测试数为 0 的"伪通过"）
  - 测试阶段 passed == total_tests
  - 代码覆盖率 >= 80%（新增代码）
验证方式：
  - API 调用获取 CI 执行结果
  - 解析测试报告 JSON/XML
```

### QG-6：集成测试门禁

```
条件：
  - 所有集成测试用例通过
  - 核心链路接口 2XX 响应
  - 响应体结构符合预期
  - 响应时间在 P99 阈值内
验证方式：
  - MCP 工具调用接口验证
  - 测试报告解析
```

### QG-7：部署验证门禁

```
条件：
  - 服务健康检查接口返回 200
  - 关键指标正常（CPU < 80%, 内存 < 85%, GC 正常）
  - 核心业务流程可正常执行
验证方式：
  - HTTP 健康检查端点
  - 监控系统 API 查询
```

### QG-8：灰度发布门禁

```
条件：
  - 灰度期间业务错误率 < 基线值 × 1.5
  - 灰度期间无 P0/P1 级别报警
  - 灰度期间核心接口 RT 无显著增加
  - 灰度观察期时长 >= 配置值
验证方式：
  - 监控系统指标查询
  - 报警系统 API 查询
```

## 门禁执行规则

### 门禁触发时机

| 门禁 | 触发阶段 | 通过后 | 不通过后 |
|------|----------|--------|----------|
| QG-1 | 阶段 1 → 阶段 2 | 进入需求评审 | 继续完善 spec.md |
| QG-2 | 阶段 2 → 阶段 3 | 进入编码实现 | 修改后重评或升级人工 |
| QG-3 | 阶段 3 → 阶段 4 | 进入编码评审 | 回退到编码实现 |
| QG-4 | 阶段 5 → 阶段 6 | 进入 CI | 修复测试 |
| QG-5 | 阶段 6 → 阶段 7 | 进入集成测试 | 按失败类型回退 |
| QG-6 | 阶段 7 → 阶段 8 | 进入部署验证 | 按失败原因回退 |
| QG-7 | 阶段 8 → 阶段 9 | 进入灰度发布 | 修复后重新部署 |
| QG-8 | 阶段 9 → 阶段 10 | 进入交付确认 | 回滚或延长灰度 |

### 门禁执行协议

```
1. 读取门禁定义（本文件）
2. 收集验证所需数据
3. 逐个检查条件是否满足
4. 所有条件均满足 → PASS
5. 任一条件不满足 → FAIL + 记录失败原因 + 执行回退路由
6. 将门禁结果写入 summary.md
```

### 例外处理

- 跳过门禁需要人工明确授权
- 授权必须在变更文档中记录：原因、授权人、时间
- 未授权的门禁跳过 = 流程违规
